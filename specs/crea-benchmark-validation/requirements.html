Summary

This feature delivers two coupled capabilities for NOXUS:

  1. Benchmark ingestion. Fetch the physical-output benchmark for the Tangshan steel cluster — the weekly blast-furnace operating rate — from CREA's public, WIND-sourced Google-Sheet CSV; clean its placeholder zeros; persist a dated, versioned raw snapshot; and emit a tidy, validated time series in parquet.
  2. Activity-signal validation. Test, transparently, whether the attributed Tangshan NO₂ signal leads that benchmark, after removing seasonality, meteorological confounding, and policy-curtailment confounding. The headline test is out-of-sample forecast skill (an AR baseline vs. AR+NO₂, compared with Diebold–Mariano), not in-sample correlation. A null result is a valid, reportable outcome.

This is the realization of the noxus/data/ (benchmark side) and noxus/validation/ modules. The NO₂ attribution that produces the predictor series is an upstream dependency (see External dependencies); this feature consumes it through a defined contract and does not implement attribution.

Cloud-gap handling for the NO₂ predictor follows the project decision of 2026-06-13 (aggregation + minimum-coverage threshold as core; reconstruction only as robustness) recorded in decisions/architecture-decisions.md.

Functional requirements

EARS format. IDs are stable references for tasks and tests.

Benchmark ingestion

REQ-001 When the benchmark ingestion runs, the system shall fetch the CREA CSV from the configured source URL and persist the unmodified bytes to data/raw/benchmark/crea_wind_<YYYY-MM-DD>.csv as a dated snapshot.
REQ-002 When parsing the CREA CSV, the system shall skip the metadata header rows (#ERROR!, Name, Frequency, Unit, ID, Time Period, Source, Update) and read the dated observation rows that follow.
REQ-003 When extracting the benchmark, the system shall select the column China: Tangshan: Operating Rate of Blast Furnaces by its exact header name (not by positional index) and the date column, producing a two-column series.
REQ-004 While cleaning the benchmark series, the system shall treat values of 0.00 (and exact zeros) as missing (NA), because a 0% blast-furnace operating rate is a not-reported placeholder rather than a real observation.
REQ-005 When emitting the benchmark, the system shall write a tidy series to data/derived/benchmark_tangshan_bf_operating_rate.parquet with columns date (date), value (float, percent), source and snapshot_date, sorted ascending by date and de-duplicated on date.
REQ-006 When the benchmark is emitted, the system shall additionally retain the configured set of auxiliary control series present in the same CSV (e.g. daily pig-iron output, daily crude-steel output) for downstream robustness use, applying the same zero-as-missing rule.

Alignment & preprocessing

REQ-010 When validation runs, the system shall load the NO₂ predictor series (with its per-period valid-coverage fraction) and the benchmark series and align them to a single common frequency (default: weekly), using a configured, documented aggregation rule.
REQ-011 While preprocessing each series, the system shall remove seasonality (configurable: STL decomposition or calendar dummies) and shall record the deseasonalization method used in the run output.
REQ-012 When preprocessing, the system shall test each series for stationarity (ADF and KPSS) and shall report the test statistics and the chosen modelling track (levels vs. differences vs. cointegration) in the run output.
REQ-013 When constructing the NO₂ predictor for modelling, the system shall residualize it against configured meteorological covariates (e.g. ERA5 boundary-layer height, wind speed, temperature) and shall include policy-curtailment periods as exogenous controls, so the signal tested is net of meteorology and known curtailments.

Lead/lag & modelling

REQ-020 When validation runs, the system shall compute the cross-correlation function between the processed NO₂ predictor and the processed benchmark over a configured lag window and report the lag of peak correlation with confidence bounds.
REQ-021 When validation runs, the system shall perform a Granger-causality test (NO₂ → benchmark) over the configured lag orders, correcting reported p-values for multiple comparisons across lags.
REQ-022 When validation runs, the system shall fit the model track selected per REQ-012 (a distributed-lag/VAR model for stationary series, or a VECM/ECM when series are cointegrated) and report coefficients with confidence intervals.
REQ-023 When evaluating leading-indicator value, the system shall run a pseudo-real-time out-of-sample evaluation (expanding or rolling window) comparing an AR-only baseline of the benchmark against an AR+lagged-NO₂ model, and shall report out-of-sample RMSE for both and the Diebold–Mariano test of equal predictive accuracy.
REQ-024 When all tests complete, the system shall emit a machine-readable results artifact (parquet/JSON) and a human-readable summary stating, in plain language, whether the NO₂ signal leads the benchmark out-of-sample, including the case where it does not (the null).
REQ-025 When a robustness run is requested, the system shall re-run the validation with the reconstructed (gap-filled) NO₂ predictor and report whether conclusions change, keeping reconstruction strictly out of the default (production) path per the 2026-06-13 decision.

CLI / orchestration

REQ-030 When the user runs the benchmark ingestion and validation via the noxus CLI, the system shall expose them as documented subcommands with configurable inputs, and shall use values from noxus/config/ for source URL, AOI, frequency, lag window, coverage threshold, and covariate set.

Non-functional requirements

NFR-001 Reproducibility. Every reported result shall be reproducible from a committed snapshot of the benchmark plus the recorded configuration; runs shall not depend on live-fetching a mutable remote at analysis time.
NFR-002 Public/free data only. All inputs shall remain public and free (project constraint); the commercial provenance of the benchmark (WIND / China United Steel Network, republished by CREA) shall be documented in references.bib.
NFR-003 Honest reporting. The summary shall report effect sizes with confidence intervals and out-of-sample skill; it shall not present an in-sample correlation alone as evidence of a leading indicator.
NFR-004 Determinism. Given identical inputs and config, statistical outputs shall be deterministic (fixed seeds where any randomness exists).
NFR-005 Stack conformance. Implementation shall use the project stack (Python ≥3.12, numpy/pandas/xarray/pyarrow) and pass uv run ruff check . and uv run pytest.

Edge cases

EDGE-001 The Tangshan column reports on a different cadence than other columns, so its rows are interleaved with 0.00 placeholders on dates belonging to other (e.g. daily) series — these must become NA and not enter aggregation as zeros (REQ-004).
EDGE-002 Numeric cells carry trailing whitespace (e.g. "96.81 "); parsing must strip and coerce robustly.
EDGE-003 The CSV column order or set may change over time; selection by exact header name guards against silent column drift (REQ-003). A missing expected column is an error (ERR-002), not a silent skip.
EDGE-004 Frequency mismatch: benchmark is weekly, NO₂ overpasses are daily-with-gaps; some weeks may fall below the minimum NO₂ coverage threshold and must be marked missing rather than aggregated from too few pixels.
EDGE-005 Short overlapping sample (TROPOMI from 2018) limits feasible VAR/lag orders; the system must cap lag orders to keep degrees of freedom positive and report when it does so.
EDGE-006 Non-stationary series with a common trend risk spurious regression; the modelling track must branch on the stationarity/cointegration result (REQ-012, REQ-022).

Error states

ERR-001 If the source URL is unreachable or returns non-CSV/empty content, the system shall fail with a clear error and shall not overwrite or corrupt an existing good snapshot.
ERR-002 If the expected Tangshan column header is absent from the fetched CSV, the system shall raise a descriptive error naming the expected header and listing the headers found.
ERR-003 If, after cleaning, the overlapping NO₂/benchmark sample is shorter than a configured minimum length, the system shall refuse to run the statistical tests and report insufficient data rather than emit unstable estimates.
ERR-004 If a required input artifact (NO₂ predictor parquet or meteorological covariates) is missing, the system shall fail with an actionable message identifying the missing dependency.

Acceptance criteria

AC-001 Running the benchmark ingestion on a committed sample CSV produces the tidy parquet with zeros converted to NA, dates parsed, sorted and de-duplicated, covering at least the 2018–present overlap window.
AC-002 A dated raw snapshot is written under data/raw/benchmark/ and is the input of record for the run (re-runs from the snapshot reproduce the parquet byte-for-byte under the same config).
AC-003 The validation run emits CCF peak-lag, Granger results (multiplicity-corrected), the selected model's coefficients with CIs, and the out-of-sample AR vs. AR+NO₂ RMSE with a Diebold–Mariano statistic.
AC-004 The human-readable summary states the lead/no-lead conclusion explicitly, reports the null when present, and lists the deseasonalization method, stationarity decision, covariates removed, lag window, and coverage threshold used.
AC-005 A robustness run with the reconstructed NO₂ predictor produces a comparison showing whether the conclusion is stable, without the reconstructed series entering the default path.
AC-006 uv run pytest and uv run ruff check . pass.

External dependencies and APIs

  • CREA benchmark CSV — public Google-Sheet CSV export (docs.google.com/spreadsheets/d/1-LumT7h3GSryFLFYzxK_QedmhI6X4y58/export?format=csv), republishing WIND / China United Steel Network steel series. Validated reachable and containing the Tangshan column, 2013-12-20 → present, weekly, on 2026-06-13.
  • NO₂ predictor (upstream NOXUS attribution) — consumed via a parquet contract: per-period attributed NO₂ over the Tangshan AOI plus a valid-coverage fraction. This feature does not implement attribution; if the artifact does not yet exist, see open-questions.html Q1.
  • Meteorological covariates — ERA5 (or equivalent public reanalysis) fields for de-confounding (REQ-013); ingestion path/contract to be confirmed (open question).
  • Policy-curtailment calendar — Tangshan production-curtailment periods (from CREA/Mysteel reporting) as an exogenous control.
  • Python libraries — statsmodels (ADF/KPSS, VAR/VECM, Granger), plus existing numpy/pandas/pyarrow. Adding statsmodels is proposed (open question Q3).
