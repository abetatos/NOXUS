Summary

Paradigm shift (developer-directed 2026-06-13). NOX-003/003.1 showed the satellite NO₂ over the Tangshan steel cluster is a weak continuous tracker of the blast-furnace operating rate (monthly r ≈ 0.14–0.19, below bar) — its variance is dominated by the slow emission-intensity decline, meteorology and non-steel background. This task stops asking for a time-series correlation and instead uses the signal as a catalyst / event marker: detect discrete production events — sharp surges/drops in cluster NO₂ (curtailment start/end, sudden shutdowns, restarts) — and test whether they (a) lead the official production print and (b) precede moves in steel-exposed markets.

Why this fits the signal: a catalyst lives in the tails (large deviations), not the bulk, so a low overall r is irrelevant; Tangshan's events are discrete and sharp (policy-driven curtailments produce step changes large versus noise); and the slow decoupling trend that killed the continuous correlation does not create sudden jumps, so it is not a confounder for event detection (it is removed once, via the NOX-003.1 residual).

Pipeline: NOX-003.1 intensity-detrended + meteo-normalised footprint residual → event detection (anomaly + change-point, coverage-screened) → ground-truth production events (CREA BF-rate jumps × public curtailment calendar) → event-study lead/lag & precision/recall vs production → market event-study (abnormal returns around events) for Chinese ferrous futures, global miners, and a China steel ETF → catalyst report.

Honesty (carried over): the headline metrics are precision / recall / lead time / cumulative abnormal return with CIs, never a cherry-picked event; a rigorous NULL (NO₂ events neither lead production nor move the asset beyond chance) is a valid, designed-for outcome (Morris & Zhang 2019). Strict no-look-ahead discipline (causal detection; trade at the next session after the overpass+latency) and multiple-testing transparency apply throughout.

Data constraint (open question Q1): the project requires public/free, reproducible data. Global miners (BHP/RIO/VALE) and a steel ETF are freely available; Chinese ferrous futures (SHFE rebar, DCE iron ore / coking coal) lack a clean free API — their daily settlements are exchange-published but ingestion is best-effort/snapshot. v1 leads with the free instruments; ferrous-futures access is resolved at approval.

Functional requirements

EARS format. IDs are stable references for tasks and tests.

NO₂ event detection

REQ-001When the catalyst pipeline runs, the system shall load the intensity-detrended, meteo-normalised footprint residual from NOX-003.1 (steel_intensity_decomposition.parquet → residual_activity, or recompute it via the index pipeline) together with the per-period valid_coverage.
REQ-002When detecting events, the system shall flag NO₂ production events as statistically significant deviations of the residual using a configurable detector set — a robust anomaly score (median/MAD z, default |z| ≥ 2.0) and a change-point / level-shift detector (e.g. CUSUM or PELT) — recording for each event: period/date, direction (surge | drop), magnitude (signed residual + z), and which detector fired.
REQ-003While detecting, the system shall coverage-screen each candidate event: a period below the inherited minimum valid-coverage is discarded (a cloud-driven gap/drop is not a production event) and never interpolated (REQ-004 of NOX-002b).
REQ-004When detecting, the system shall operate on the meteorology-normalised series so a ventilation-driven spike (stagnant air) is not counted as a production event, and shall record that the meteo control was applied (it is more critical for short-window events than for monthly means).
REQ-005When normalising meteorology for event detection, the system shall strengthen the meteo control beyond NOX-003's wind+PBL regress-out with transport/stagnation-aware covariates derived from the existing ERA5 snapshot — a ventilation index (≈ 10 m wind speed × boundary-layer height) and an atmospheric-stagnation flag (low wind × shallow PBL) — and shall, for any candidate event, verify it survives this control (a deviation explained by a stagnation day is rejected). Rationale (developer-directed 2026-06-13): the cheap, in-framework lever for "better weather correlation" on short windows is a stronger meteo normalisation, not the full flux-divergence method (deferred, NOX-005); a still-air week, not a production change, is the dominant false-positive source for an event marker.

Ground-truth production events

REQ-010When labelling, the system shall derive operating-rate events from the CREA blast-furnace operating-rate benchmark (NOX-001) as large weekly changes (|Δ| or rolling robust-z above a configurable threshold), direction-typed (up | down).
REQ-011When labelling, the system shall ingest a best-effort public curtailment calendar (MEE / CREA reporting: heating-season cuts, blue-sky/summit curtailments, sudden environmental shutdowns) as discrete production-event intervals (start, end, direction), documenting the source; absence is handled explicitly (assumption A4 / NOX-003 Q6a).
REQ-012When labelling, the system shall combine the two into a unified production-event set (a benchmark jump and/or a calendar episode) and tag each with its cause where known (policy vs market), per the developer decision to use both sources.

Event study vs production

REQ-020When evaluating, the system shall match NO₂ events to production events within a configurable temporal window and compute, as the headline metrics, precision (matched NO₂ events / all NO₂ events), recall (matched production events / all production events), the false-positive rate, and the signed lead time (NO₂ event date − production event date) with a confidence interval / sign test — not a Pearson r.
REQ-021When reporting, the system shall state explicitly whether NO₂ events lead the official BF-rate print (positive median lead beyond the publication latency) — the timeliness edge — and quantify it, accepting a non-leading / null result as reportable.

Market linkage

REQ-030When the market study runs, the system shall ingest daily prices for the configured instrument families from public/free sources and persist dated snapshots (reproducible, no live fetch at analysis): global miners (BHP, RIO, VALE) and a China steel / materials ETF via a free API; Chinese ferrous futures (SHFE rebar, DCE iron ore, coking coal) via the exchange-published daily settlement where a free path exists, else a documented best-effort snapshot (Q1).
REQ-031When studying market impact, the system shall compute returns and abnormal returns (vs a market/sector benchmark via a market model or simple de-meaning) over a configurable window [−k, +k] trading days around each NO₂ event, emitting per-event and aggregated cumulative abnormal return (CAR).
REQ-032When reporting market impact, the system shall report catalyst metrics: directional hit rate (did the asset move the expected way after the event), mean/median abnormal return with a CI, and the lead versus the official print — never a single event.
REQ-033When studying, the system shall separate production-confirmed NO₂ events from unconfirmed ones; the tradeable hypothesis is on confirmed events (or on NO₂ as an early flag of soon-to-be-confirmed events), and the report shall distinguish the two.

Reporting, no-look-ahead, multiplicity

REQ-040When the pipeline completes, the system shall emit a machine-readable event table + a plain-language catalyst report that states the NULL explicitly when present (NO₂ events neither lead production nor move the asset beyond chance) as a valid outcome, with precision/recall/lead/CAR reported with confidence intervals.
REQ-041The system shall enforce no look-ahead: event-detection baselines/thresholds at time t use only data up to t (causal), and the market study aligns each event to the next tradeable session after the overpass + processing latency, so no future information enters the signal.
REQ-042The system shall apply multiple-testing discipline: record the number of events, windows, thresholds and instruments tested and adjust or flag the multiplicity, so a positive CAR is not a product of search (no p-hacking over knobs).

CLI / orchestration

REQ-050When the user runs the catalyst via the noxus CLI, the system shall expose documented subcommands (e.g. detect-events, ingest-market, catalyst) driven by a CatalystConfig (detector thresholds, match window, event-study window, instruments, latency, paths); existing NOX-001/002/003 subcommands are unchanged.

Non-functional requirements

NFR-001Reproducibility. Every result reproduces from committed NO₂ residual + benchmark + dated market snapshots + recorded config; no live fetch at analysis time.
NFR-002Public/free data only. Market data from free sources; the Chinese-ferrous-futures tension is resolved at approval (Q1) — never a paywalled dependency without explicit developer sign-off.
NFR-003No look-ahead / causal. Detection and trade-alignment use only information available at the decision time, including the TROPOMI processing latency.
NFR-004Honest reporting. Precision/recall/lead/CAR with CIs and multiplicity; the null is first-class; no cherry-picked event or instrument.
NFR-005No-publish. NO₂-derived event series are gitignored; market snapshots are third-party data (gitignored unless licence permits); the small stats summary may be committed.
NFR-006Determinism + stack conformance. Fixed seeds; Python ≥3.12; passes uv run pytest and uv run ruff check .; any new dependency (market data, change-point) needs developer approval (Q2/Q3).

Edge cases

EDGE-001Two NO₂ events fall inside one match window → both are matched to the nearest production event without double-counting; reported.
EDGE-002A heating-season blanket curtailment (long interval) vs a spot event → interval episodes are matched on their start/end edges, not every period inside, to avoid inflating recall.
EDGE-003An NO₂ event window straddles market holidays → the study uses the next available trading sessions; the window is in trading days, not calendar days.
EDGE-004An instrument lacks history for an early event → that event is dropped for that instrument (reported), not back-filled.
EDGE-005The event period is cloud-gapped (low coverage) → not an event (REQ-003); a true curtailment hidden by cloud becomes a (reported) missed detection, not a false negative silently absorbed.
EDGE-006No NO₂ events clear the threshold on the (short, partial) series → the pipeline reports "insufficient events for a stable study" rather than emitting unstable rates.

Error states

ERR-001If the NOX-003.1 residual or the NOX-001 benchmark is missing, the system fails with an actionable message naming the producing command.
ERR-002If market data for a requested instrument cannot be fetched, the system skips that instrument with a recorded note (and continues with the others) rather than failing silently or fabricating prices.
ERR-003If fewer than a configured minimum number of (confirmed) events exist after screening, the system refuses the statistical study and reports insufficient events (cf. EDGE-006).

Acceptance criteria

AC-001On a planted-event synthetic residual (known step surges/drops at known dates) the detector recovers the event dates and directions within tolerance; precision/recall against the planted truth are computed correctly.
AC-002On a planted-null residual (no events, pure noise) the false-positive rate is low (≤ the configured |z| tail), proving the detector does not invent events.
AC-003Coverage-screened (cloud-gap) and meteo-spike synthetic periods are not flagged as events (REQ-003/004).
AC-004Production-event labelling recovers planted BF-rate jumps and calendar intervals; matching computes the correct lead time on a planted lead.
AC-005The market event-study recovers a planted cumulative abnormal return around events on synthetic returns; the directional hit rate and CI are correct; a planted-null (returns independent of events) yields a CAR indistinguishable from zero.
AC-006No-look-ahead guard: a test proves detection at time t is unchanged by future data, and the market alignment never uses a session before the overpass+latency.
AC-007The ferrous-data-absent path degrades gracefully (ERR-002): the study completes on the available instruments with a recorded skip.
AC-008The catalyst report states the conclusion in plain language including the null, with precision/recall/lead/CAR + CIs and the multiplicity count; uv run pytest and uv run ruff check . pass.
