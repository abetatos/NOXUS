Design — TROPOMI NO₂ acquisition + AOI subset + QA

Task ID NOX-002a
Feature slug tropomi-no2-acquisition
Status spec_draft

Technical summary

A resumable acquisition loop that, per time-batch, submits an openEO job to CDSE which loads the Sentinel-5P L2 NO₂ collection, filters spatially to the AOI bbox and temporally to the batch, applies the QA mask server-side, and returns only the AOI window. Each returned overpass is written to a per-overpass file under data/raw/tropomi/ with a provenance sidecar (collection, processor version, QA threshold, AOI, fetch time). A lightweight manifest records what has been fetched so re-runs are idempotent and resumable. Authentication reuses the CDSE account from .env (the openEO client reads OIDC config from environment; interactive device-flow for dev, client-credentials for headless).

A separate verification module reads the acquired store, selects clear-sky high-NO₂ days, and renders the NO₂ field over the AOI with facility points overlaid (optionally an optical thumbnail). Acquisition keeps L2-per-overpass detail; gridding/compositing is deferred to NOX-002b.

Security and permissions

Credentials (CDSE_*) are read from .env via python-dotenv; never logged, printed, or committed. Network egress limited to CDSE openEO (and optional S2 for verification). No protected files written, no remote mutations.

docs/data-access.md (openEO acquisition how-to), README.md (Status/Data sources), project map (data/tropomi.py implemented, new raw store). Reviewer confirms.

Test design

  • Mocked openEO: a fake connection returning a synthetic AOI overpass (xarray) so the subset→QA→persist→manifest path is tested without network/credentials (AT1–AT4).
  • QA + coverage: synthetic window with known qa distribution → filter and coverage fraction correct (AT3).
  • Resumability: run twice; second run fetches nothing new (AT4).
  • Version tracking: synthetic overpasses tagged v1/v2 → discontinuity flagged (AT2).
  • Verification: given a synthetic store, clear-day selection picks the low-cloud/high-NO₂ day and render_day writes a PNG with facility points (AT5; rendering asserted by file existence + a few plotted markers, real-data visual check is manual).
  • Errors: auth failure, collection-not-found, job failure/resume (AT-ERR-1).

UI verification plan

None (no browser UI). The NO₂/facility renders are inspected manually as the location/sanity check.

Risks and trade-offs

  • openEO collection semantics. Whether the CDSE openEO S5P collection already harmonises OFFL/RPRO and exposes the processor version is unconfirmed → open Q1; design isolates the load step so this can be set without rework.
  • Credit budget. Processing 3,734 dates may exceed 10k credits/month → batch by year/month; acquisition is resumable so it can span billing cycles.
  • Sub-pixel facilities. Signal is cluster/sub-cluster (assumptions A1); verification sets the right expectation rather than promising per-plant detail.
  • Rejected: full-granule download (2.2 TB), Sentinel Hub/GEE (L3 rasters, less L2 control; SH needs extra OAuth client) — see decisions/architecture-decisions.md (openEO decision).

Rollback plan

Additive: new module behind new CLI subcommands. Rollback = revert the branch and delete data/raw/tropomi/ (gitignored) and the verification PNGs. If openeo is problematic, remove the dep and disable the subcommands; nothing downstream exists yet.
