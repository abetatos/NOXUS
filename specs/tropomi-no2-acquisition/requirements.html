Summary

Acquire the predictor substrate: Sentinel-5P/TROPOMI L2 tropospheric NO₂ over the Tangshan area of interest (AOI), for the TROPOMI era (2019-01 → present), via openEO server-side spatial subsetting on the Copernicus Data Space Ecosystem (CDSE). Downloading full orbit granules is infeasible (~3,734 overpasses × ~609 MB ≈ 2.2 TB); openEO returns only the AOI window, keeping ~0.2–1 GB. Use a version-consistent record (OFFL + RPRO) and track the processor version per observation. Apply the recommended quality filter, retain the analysis-relevant L2 variables (incl. the averaging kernel), track per-period valid coverage, and make the acquisition resumable.

This task ends at a per-overpass, AOI-clipped, QA-filtered NO₂ store. Gridding / temporal compositing is NOX-002b; attribution to facilities is NOX-003. A verification step (clear-sky, high-NO₂ days, NO₂ field with facility overlay) is included to confirm the location and that the signal is physically sensible.

Scope reality (see assumptions.html): at TROPOMI's ~5.5×3.5 km footprint a steel plant is sub-pixel and 14 of 26 operating plants share a pixel with a neighbour, so the signal is cluster / sub-cluster level, not per-factory.

Functional requirements

Acquisition

REQ-001When acquisition runs, the system shall request Sentinel-5P L2 tropospheric NO₂ over the configured AOI and date range via openEO with a server-side spatial subset (AOI bbox), never downloading full orbit granules.
REQ-002When acquiring, the system shall use a version-consistent record (OFFL + RPRO where available) and shall record the processor/algorithm version for each observation so the v1→v2 discontinuity can be detected and flagged.
REQ-003While ingesting, the system shall apply a configurable quality filter, default qa_value ≥ 0.75, discarding pixels below the threshold.
REQ-004When ingesting, the system shall retain, per overpass over the AOI: nitrogendioxide_tropospheric_column, qa_value, cloud fraction, the averaging kernel, geolocation (pixel centres/corners), and the column precision/uncertainty.
REQ-005When an overpass is ingested, the system shall persist the clipped result reproducibly under data/raw/tropomi/, tagged with date, AOI, QA threshold, collection and processor version (provenance).
REQ-006When ingesting, the system shall compute and store a valid-coverage fraction per overpass over the AOI (share of AOI pixels passing QA), feeding the cloud-handling policy (2026-06-13 decision).
REQ-007While acquiring the full record, the system shall be idempotent and resumable: re-running skips overpasses already fetched and continues after interruption, and shall support batching to respect the openEO credit budget.

Verification

REQ-010When verification runs, the system shall select clear-sky, high-signal days (low cloud fraction and high AOI NO₂) from the acquired record.
REQ-011For a selected day, the system shall render the TROPOMI NO₂ field over the AOI with the steel-facility locations overlaid, so a reviewer can confirm the NO₂ enhancement co-locates with the steel sub-clusters.
REQ-012The verification shall optionally include an optical thumbnail (e.g. Sentinel-2 true colour) of a facility location, as an independent check that a plant is physically present where the NO₂ enhancement appears.

CLI

REQ-020When the user runs the acquisition and verification via the noxus CLI, the system shall expose them as documented subcommands driven by noxus/config/ (AOI/buffer, date range, QA threshold, collection/version, credit-batch size).

Non-functional requirements

NFR-001Reproducibility. Acquisition is parameterised and provenance-tagged; given the same config and source state, the kept dataset is reproducible. Raw clips live under gitignored data/raw/.
NFR-002Public/free. Uses the existing CDSE account via openEO within the free credit quota; credentials are read from .env per .env.example, never logged or committed.
NFR-003Volume discipline. Only the AOI window is transferred; no full-granule downloads.
NFR-004Stack conformance. Python ≥3.12; passes uv run ruff check . and uv run pytest; the openeo package is added (geo I/O may use the geo extra).

Edge cases

EDGE-001Heavily clouded periods yield few/no valid pixels → low coverage; the overpass is kept with its coverage flag, not interpolated (cloud decision).
EDGE-002Processor-version discontinuity (v1→v2) within the record → must be tracked (REQ-002) and surfaced, not silently mixed.
EDGE-003openEO jobs fail/time out or hit credit limits → resumable batching (REQ-007); partial progress is not lost.
EDGE-004Some days have two overpasses over the AOI (swath overlap at this latitude) → both retained, de-duplicated by acquisition time.
EDGE-005An AOI request returns an empty/all-masked window → recorded as zero-coverage, not an error.

Error states

ERR-001If CDSE/openEO authentication fails (missing/invalid credentials), the system shall fail with a clear message referencing the .env variable names — never echoing secret values.
ERR-002If the openEO collection or NO₂ band cannot be found, the system shall raise a descriptive error naming the collection/band attempted.
ERR-003If an openEO job fails irrecoverably for a batch, the system shall record the failure, leave already-fetched overpasses intact, and allow a resume.

Acceptance criteria

AC-001A small date-range acquisition over the AOI returns AOI-clipped NO₂ (with qa/cloud/kernel/geo), persisted with provenance, transferring only the AOI window.
AC-002Each stored overpass carries its processor version and a valid-coverage fraction; a mixed-version range surfaces the discontinuity.
AC-003Re-running the acquisition skips already-fetched overpasses (idempotent/resumable).
AC-004The verification produces, for a clear high-NO₂ day, a NO₂-over-AOI render with facility points overlaid showing enhancement at the steel sub-clusters (optionally with an optical thumbnail).
AC-005uv run pytest and uv run ruff check . pass; auth/network are mocked in tests.

UI acceptance criteria

None (no browser UI). Verification output is static image/plot files inspected by a human.

Visual evidence

The verification renders (NO₂ field + facility overlay; optional optical thumbnail) are produced by this feature as evidence, not inputs. No input screenshots.

External dependencies and APIs

  • CDSE openEO — Sentinel-5P L2 NO₂ collection; auth via the existing CDSE account (CDSE_* in .env); 10,000 free credits/month. Python package openeo (new).
  • Facilities CSV — data/derived/tangshan_steel_facilities.csv for the verification overlay and AOI provenance.
  • AOI — noxus.config.region.TANGSHAN / tangshan_aoi(buffer).
  • (Optional) optical basemap — Sentinel-2 true colour (via openEO/Sentinel Hub or a static tile) for REQ-012; source TBD (open question).
