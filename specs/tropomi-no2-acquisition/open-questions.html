Q1openEO S5P collection: id, OFFL/RPRO harmonisation, processor-version exposure, L2 detailResolved
Question
What is the exact CDSE openEO collection id for S5P L2 NO₂, does it harmonise OFFL/RPRO, does it expose the processor version, and does the subset return L2-level detail (incl. averaging kernel) or a pre-gridded product?
Findings (verified live 2026-06-13)
Collection id is SENTINEL_5P_L2; bands include NO2 and CLOUD_FRACTION (+ dataMask) but no qa_value band and no averaging kernel — it is a gridded product with quality screening applied internally. The collection serves one band per request (NO2 and CLOUD_FRACTION fetched separately and merged). The time dimension is named t; openEO marks it unlimited (handled on write). The processor version is not exposed as a field → derived from date (fallback cutoff).
Consequences
(1) Client-side qa_value filtering is not possible; we rely on the collection's internal screening and use cloud_fraction for clear-day selection/coverage. (2) The averaging kernel is not available via openEO → deferred to a future L2 path (CDSE OData/S3) for fine attribution (NOX-003). REQ-004's kernel retention is downgraded to "as exposed by the collection".

Q2Headless auth vs interactiveResolved
Question
Run openEO with interactive device-flow (just the account) or set up a non-interactive OIDC client (CDSE_CLIENT_SECRET) for unattended batch acquisition?
Why it matters
Acquiring ~3,734 overpasses may run long/unattended.
Decision (2026-06-13, developer)
Use the account credentials already in .env: mint a CDSE token via the OIDC password grant (CDSE_USERNAME/CDSE_PASSWORD, client_id=cdse-public) and pass it to openEO via authenticate_oidc_access_token(). If that fails, fall back to authenticate_oidc() (refresh token, else interactive device flow that opens the web). No separate OAuth client required.
