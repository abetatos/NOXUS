Summary

Acquire a diurnal NO₂ predictor substrate for Tangshan from GEMS (Geostationary Environment Monitoring Spectrometer, on GEO-KOMPSAT-2B / GK2B, KR), the first geostationary air-quality sounder. GEMS makes ~8 daytime scans/day over East Asia (vs TROPOMI's single ~13:30 overpass), which resolves the diurnal / day-of-week cycle — the one temporal lever (short of flux divergence, NOX-005) that can lift steel's variance share above the ~40% mean-share SNR ceiling, by separating flat steel baseload from the cyclical traffic term. NOX-006 confirmed this lever is invisible to TROPOMI (weekly component flat, 0.5% variance) precisely because of the single overpass.

This task delivers a per-scan, AOI-clipped, QA-filtered GEMS L2 NO₂ store under data/raw/gems/ with diurnal structure (multiple timestamps per day), provenance, and a valid-coverage fraction — the GEMS analogue of NOX-002a. Diurnal/weekly compositing, source-separation analysis and re-running NOX-003.1/NOX-004/NOX-006 on the GEMS residual are downstream follow-ups (future NOX-009), not in scope here.

Scope decision (2026-06-14): a few stable-weather months, not the full mission. The purpose is to test whether a weekly component exists at all, so the default acquisition scope is a climatologically-stable autumn window (September–October) rather than the full ~Oct-2021→present record — see open-questions.html Q5 / assumptions.html A7. Early autumn (post-monsoon, pre-heating-season) gives near-constant meteorology (low synoptic noise → the weekly cycle is not swamped), no residential-heating NO₂ confounder (so steel+traffic dominate the column → cleaner source separation), low cloud (balanced GEMS coverage across all weekdays and ~8 daily scans), and a good sun angle at 39.6°N. Primary window: Sep–Oct 2023 (overlaps the strongest TROPOMI-era regime); optional replication on Sep–Oct 2024 (same season, different year → controls climate while adding robustness). This bounds the download to ~50–150 GB/window (REQ-009). Full-mission acquisition is opt-in.

Central viability constraint (verified, see assumptions.html A1 / open-questions.html Q1). Unlike TROPOMI (openEO filter_bbox) and ERA5 (CDS area), the GEMS distribution channels — NIER-ESC Open-API, SFTP, and no-login web download — deliver full granules (the full ~5000×5000 km field of regard). There is no documented server-side AOI/bbox request. We therefore download full granules and clip the Tangshan AOI client-side. The kept volume after clip is tiny (a ~0.5° AOI ≈ a few dozen pixels/scan), but the download transit volume is the cost and its magnitude is the feasibility unknown — so a measure-one-granule spike (T1) gates the build (REQ-000).

Scope reality: GEMS' nominal ~3.5 km (N/S) × ~8 km (E/W) footprint is near nadir (Korea); the L3 product is documented at ~5 km over Korea and ~10 km elsewhere, so over Tangshan — off-nadir (sub-satellite point 128.0°E; Tangshan 39.6°N, 118.4°E) and near the 45°N field-of-regard edge — the effective pixel is ~10 km. A steel plant is still sub-pixel, so the signal stays cluster / sub-cluster level, not per-factory, consistent with the TROPOMI work. GEMS' gain over TROPOMI is diurnal frequency, not spatial resolution.

Functional requirements

Viability gate (spike)

REQ-000Before building the acquisition loop, the system shall (T1 spike, developer-run with the issued key) download one real GEMS L2 NO₂ granule and record: (a) file size on disk and the implied per-day (×~8 scans) and full-mission volume; (b) the granule's spatial footprint, array dimensions and geolocation; (c) the effective pixel size over Tangshan (off-nadir); (d) which NO₂ product fields exist (tropospheric vs total column, QA flag, averaging kernel, precision) and the processor version. A go/no-go decision against a download-budget threshold (Q1/A2) shall be recorded in open-questions.html and the decision log before further implementation.

Acquisition

REQ-001When acquisition runs, the system shall enumerate the GEMS L2 NO₂ granules intersecting the configured date range via the NIER-ESC Open-API (getFileDateList/getFileList) and download each granule via getFileItem, then clip to the Tangshan AOI bbox client-side and discard the rest. (SFTP is the documented fallback channel; see design.)
REQ-002When acquiring, the system shall keep the diurnal structure: every retained scan is tagged with its observation timestamp (local scan slot) so the multiple daily passes are preserved distinctly (this is the whole point of GEMS over TROPOMI).
REQ-003When acquiring, the system shall pin a single product version (configurable, e.g. the current operational ver) for a version-consistent record and shall record the processor/algorithm version per scan so a version discontinuity is detectable, never silently mixed.
REQ-004While ingesting, the system shall apply a configurable quality filter using the GEMS QA flag/value (default = the product-recommended good-quality threshold; resolved at T1), discarding pixels below it.
REQ-005When ingesting, the system shall retain, per scan over the AOI: the tropospheric NO₂ column (primary) and, if present, the total column; the QA flag/value; cloud fraction; the averaging kernel (if exposed); geolocation (pixel centres/corners); and the column precision/uncertainty.
REQ-006When a scan is ingested, the system shall persist the clipped result reproducibly under data/raw/gems/, tagged with date, scan time, AOI, QA threshold, product/version and fetch time (provenance).
REQ-007When ingesting, the system shall compute and store a valid-coverage fraction per scan over the AOI (share of AOI pixels passing QA), feeding the same coverage-screen / never-interpolate policy used for TROPOMI.
REQ-008While acquiring the record, the system shall be idempotent and resumable: a manifest records fetched granules/scans; re-running skips what is done and continues after interruption; acquisition shall support batching by date window to respect the channel's session/volume limits (web: 20 GB / 40 files per session — no-login only; Open-API/SFTP is the bulk route, confirm limits at T1).
REQ-009The default acquisition scope shall be a stable-weather window (Sep–Oct, primary year 2023; optional 2024 replication), not the full mission, to test for a weekly component under near-constant meteorology (A7/Q5); full-mission acquisition is opt-in via config. The system shall record per-scan calendar metadata (weekday + a Chinese-public-holiday flag, esp. Golden Week 1–7 Oct) so the downstream weekday profile (NOX-009) can exclude/model holidays that break the 7-day cycle.

Verification

REQ-010When verification runs, the system shall select a clear-sky, high-signal scan (low cloud fraction and high AOI NO₂) from the acquired record.
REQ-011For a selected scan, the system shall render the GEMS NO₂ field over the AOI with the steel-facility locations overlaid, so a reviewer can confirm the enhancement co-locates with the steel sub-clusters (reusing the TROPOMI facilities CSV + overlay convention).
REQ-012Verification shall additionally render a diurnal panel: the AOI-mean NO₂ across the day's ~8 scans for one clear day, as direct evidence that GEMS resolves the within-day cycle TROPOMI cannot — the core justification for this acquisition.

CLI

REQ-020When the user runs the acquisition and verification via the noxus CLI, the system shall expose them as documented subcommands (e.g. fetch-gems, verify-gems) driven by noxus/config/ (AOI/buffer, date range, QA threshold, product/version, batch window).

Non-functional requirements

NFR-001Reproducibility. Acquisition is parameterised and provenance-tagged; given the same config and source state, the kept dataset is reproducible. Raw clips live under gitignored data/raw/gems/; downloaded full granules are treated as a transient cache and never committed.
NFR-002Public/free. Uses the free NIER-ESC GEMS distribution. The Open-API key is read from an environment variable (e.g. GEMS_API_KEY) via .env per .env.example — never logged or committed; reference the variable name only.
NFR-003Volume discipline. Full granules are downloaded only because no server-side subset exists; each granule is clipped to the AOI immediately and the full granule is deleted (streaming clip) unless a --keep-granules debug flag is set. Only the AOI clip is retained.
NFR-004Stack conformance. Python ≥3.12; passes uv run ruff check . and uv run pytest; downloads via the existing HTTP stack (httpx); NetCDF/HDF5 read via the existing h5netcdf/h5py (add netCDF4/geo extra only if the GEMS HDF layout requires it — confirm at T1).
NFR-005Politeness. Downloads are rate-limited / sequential with retry+backoff to respect the free service and its session limits; the run is resumable so it can span sessions.

Edge cases

EDGE-001Heavily clouded scans yield few/no valid AOI pixels → low coverage; the scan is kept with its coverage flag, not interpolated (same policy as TROPOMI).
EDGE-002Processor/version discontinuity within the record (v2.x ↔ v3.x ↔ v4.x landscape) → tracked (REQ-003) and surfaced, not silently mixed.
EDGE-003A download fails / times out / hits a session or volume limit → resumable batching (REQ-008); partial progress is not lost; the failed granule is retried later.
EDGE-004Tangshan sits near the northern edge of the field of regard (N limit ~45°N) and off the sub-satellite longitude → off-nadir, coarser pixels; a scan whose footprint does not actually cover the AOI (edge/partial scan mode) is recorded as zero-coverage, not an error.
EDGE-005Scan count per day varies (seasonal daylight, instrument duty cycle, calibration slots) → the diurnal series has a ragged number of slots per day; downstream must not assume a fixed 8.
EDGE-006An AOI clip returns an empty/all-masked window → recorded as zero-coverage, not an error.

Error states

ERR-001If the GEMS Open-API key is missing/invalid, the system shall fail with a clear message referencing the environment variable name — never echoing the key value.
ERR-002If the Open-API returns an error/empty listing for a date range, or the NO₂ product/field cannot be found in a granule, the system shall raise a descriptive error naming the product/field/version attempted.
ERR-003If a download fails irrecoverably for a batch, the system shall record the failure, leave already-fetched scans intact, and allow a resume.
ERR-004If the measured per-granule / full-mission download volume (REQ-000) exceeds the configured budget threshold, the system shall stop and surface the no-go rather than silently starting a multi-TB download.

Acceptance criteria

AC-000The T1 spike produces a recorded measurement of one real granule's size + structure + Tangshan footprint + available NO₂ fields + version, and a written go/no-go against the budget threshold.
AC-001A small date-range acquisition over the AOI returns AOI-clipped GEMS NO₂ with multiple scans per day (diurnal structure preserved), each with QA/cloud/coverage/geo and provenance, retaining only the AOI window (full granules not kept).
AC-002Each stored scan carries its product version and a valid-coverage fraction; a mixed-version range surfaces the discontinuity.
AC-003Re-running the acquisition skips already-fetched scans (idempotent/resumable); interruption mid-range resumes without re-downloading done granules.
AC-004Verification produces, for a clear high-NO₂ scan, a NO₂-over-AOI render with facility points overlaid, plus a diurnal panel showing the within-day NO₂ cycle across the day's scans.
AC-005uv run pytest and uv run ruff check . pass; network/auth are mocked in tests.

UI acceptance criteria

None (no browser UI). Verification output is static image/plot files inspected by a human.

Visual evidence

The verification renders (NO₂ field + facility overlay; diurnal panel) are produced by this feature as evidence, not inputs. No input screenshots.

External dependencies and APIs

  • NIER-ESC GEMS distribution — https://nesc.nier.go.kr. Channels: Open-API (key by application, ~7-day admin approval; operations getFileDateList, getFileList, getFileItem; JSON/XML/TEXT; params date/sDate/eDate/ver); SFTP (credentials by application, IP-bound); no-login web download (≤20 GB / 40 files per session). All deliver full granules (no AOI subset).
  • GEMS L2 NO₂ product — NetCDF4/HDF5; tropospheric + total column NO₂, QA flag, averaging kernel, precision. Version landscape v2.x→v4.0.1 (operational ver pinned at T1). Coverage from ~Oct-2021 (reprocessed/operational) over East Asia (N/S 45°N–5°S, E/W 75°E–145°E).
  • Facilities CSV — data/derived/tangshan_steel_facilities.csv for the verification overlay and AOI provenance (reused from TROPOMI).
  • AOI — noxus.config.region.TANGSHAN / tangshan_aoi(buffer) (reused).
