Design — GEMS (geostationary) NO₂ acquisition + AOI clip + QA

Task ID NOX-007
Feature slug gems-acquisition
Status blocked

Technical summary

A resumable acquisition loop that, per date-batch, (1) enumerates GEMS L2 NO₂ granules via the NIER-ESC Open-API (getFileDateList/getFileList, pinned product ver), (2) downloads each full granule via getFileItem over httpx with retry/backoff, (3) opens it lazily and clips to the Tangshan AOI bbox client-side (the granule covers the full ~5000×5000 km field of regard — there is no server-side subset), (4) applies the QA filter, computes valid-coverage, and (5) writes a per-scan AOI clip under data/raw/gems/ with a provenance sidecar, then deletes the full granule (transient cache; kept only under a debug flag). A lightweight manifest records what has been fetched so re-runs are idempotent and resumable. The Open-API key is read from an environment variable (GEMS_API_KEY); SFTP is the documented fallback channel (same enumerate→download→clip flow, different transport).

Because GEMS scans the daytime ~8×, the store is per-scan (multiple timestamps per day) — the diurnal structure that distinguishes GEMS from TROPOMI is preserved at acquisition and is the input to the downstream source-separation work (future NOX-009). A separate verification module renders a clear-scan NO₂/facility overlay and a diurnal panel (AOI-mean NO₂ across the day's scans) as direct evidence GEMS resolves the within-day cycle.

Gate. Task T1 is a measurement spike: download one real granule, measure its on-disk size + structure + Tangshan footprint + available NO₂ fields + version, and record a go/no-go against a configured download-volume budget before the loop is built. The per-granule byte size is not publicly documented and is the feasibility unknown.

Security and permissions

The Open-API key (GEMS_API_KEY) is read from .env via python-dotenv; never logged, printed, or committed (it appears in request URLs/headers, so logging of full request URLs must be scrubbed). Network egress limited to the NIER-ESC host. No protected files written, no remote mutations.

Test design

  • Mocked Open-API: a fake HTTP layer returns a synthetic granule listing + a small synthetic NetCDF granule (multiple scan slots over a region covering the AOI) so enumerate→download→clip→QA→persist→manifest is tested without network/key (AT1/AT1b).
  • QA + coverage: synthetic AOI window with a known QA distribution → filter and coverage fraction correct (AT3).
  • Diurnal preservation: a synthetic day with several scan slots → all retained with distinct timestamps; downstream sees a per-scan series (AT1b).
  • Resumability: run twice; second run downloads/clips nothing new; mid-range interruption resumes (AT4).
  • Version tracking: synthetic scans tagged with different versions → discontinuity flagged (AT2).
  • Verification: synthetic store → clear-scan selection + render_scan writes a PNG with facility markers + render_diurnal_panel writes the within-day curve (AT5a; real-data visual check manual, AT5b).
  • Errors: missing/invalid key (no value echoed), product/field-not-found, download failure/resume, budget no-go (AT-ERR-1).
  • Volume discipline: assert the full granule is deleted after clip unless --keep-granules (NFR-003).

None (no browser UI). The NO₂/facility render and the diurnal panel are inspected manually as the location/diurnal-resolution check.

Risks and trade-offs

  • Download volume (the gate). No server-side subset → full granules transit the wire; narrowing the date range is the only lever to cut it. Per-granule size unknown (order 10²–10³ MB). Bounded by the scoped-window default (see below) to ~50–150 GB/window rather than a multi-TB full mission. Mitigations: clip-then-delete (kept volume tiny), resumable batching, SFTP bulk route (the 20 GB/40-file cap is no-login-web only), run on a server-class machine with bandwidth/scratch. T1 measures and gates this (REQ-000/ERR-004). This is the reason the task is blocked: it is to be executed on a server-style machine in the future.
  • Scoped window, not full mission (REQ-009 / Q5 / A7). Default scope = a stable-weather autumn window, Sep–Oct 2023 (optional Sep–Oct 2024 replication): post-monsoon + pre-heating-season ⇒ near-constant meteorology (weekly cycle not swamped), no residential-heating NO₂ confounder (cleaner steel↔traffic separation), low cloud (balanced GEMS coverage across weekdays + the ~8 daily scans). Chinese public holidays (esp. Golden Week 1–7 Oct) are flagged per scan and excluded/modelled in the downstream weekday profile (they break the 7-day cycle). Full mission stays opt-in.
  • Shorter history. ~Oct-2021→ vs TROPOMI 2019→ — less statistical power; mitigated for this test by focusing on the diurnal/weekly signal within the scoped window rather than a long levels series.
  • Off-nadir resolution at Tangshan. Tangshan is ~9.6° west of the 128°E sub-satellite point and near the 45°N north edge of the FOR → effective pixel ~10 km (L3 documented ~5 km Korea / ~10 km elsewhere), coarser than the 3.5×8 nominal. Adequate for the cluster (≈ NOX-002b's ~5 km decision), not for per-plant. GEMS' gain is diurnal frequency, not resolution.
  • Ceiling unchanged in levels. GEMS does not change the ~40% mean-share cap on a levels estimator. The thesis is purely a variance-share gain from resolving the diurnal/weekly cycle (the NOX-006 lever). Honest-null remains valid.
  • GEMS retrieval biases at high NO₂ / high latitude (v2/v3 validation literature) → keep the relative-index discipline; signal is relative, not absolute.
  • Rejected alternatives: (a) GEMS L3 daily/monthly gridded NO₂ — far smaller, but averages away the diurnal cycle ⇒ defeats the entire purpose; (b) no-login web download as the main channel — 20 GB / 40-file session cap makes a multi-TB pull impractical; Open-API/SFTP scripted + resumable is the route; (c) external-HDD-by-mail — not reproducible/programmatic.

Rollback plan

Additive: new module behind new CLI subcommands; TROPOMI/ERA5 paths untouched. Rollback = revert the branch and delete data/raw/gems/ (gitignored) + the GEMS verification PNGs. If the T1 go/no-go is a no-go, the task stays blocked/closed with the measurement recorded and no loop is built — nothing downstream depends on it yet.
