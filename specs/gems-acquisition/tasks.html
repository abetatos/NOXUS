Implementation tasks — GEMS NO₂ acquisition

Task ID NOX-007
Feature slug gems-acquisition
Status blocked

Blocked — execution deferred

Why blocked

The acquisition downloads full granules (no server-side AOI subset) and clips client-side. The transient download volume over the full mission is plausibly multi-TB and the per-granule size is unmeasured. By developer decision the build/run is deferred to a server-style machine (bandwidth + scratch) in the future. The spec is complete so it can be executed then. Do not implement while status is blocked; the developer moves it to human_approved when ready to run.

Tasks

Implement in order; small, verifiable steps. Mark complete by adding class="done" to the <li class="task-item">. Global status lives in tasks.json. T1 is a gating spike — its go/no-go decides whether T2+ proceed.

  1. T1 GATE — measure one granule + go/no-go (REQ-000, ERR-004). (developer-run, needs the issued Open-API key + a server-class machine)
    Register the Open-API key (~7-day approval). Download one real GEMS L2 NO₂ granule via getFileItem. Record: on-disk size + implied per-day/full-mission volume; array dims + geolocation; effective pixel size over Tangshan (off-nadir); which NO₂ fields exist (tropospheric/total/QA/kernel/precision) + exact names; processor version; the Open-API response schema + per-key rate/volume limits. Write a go/no-go vs the download budget into open-questions.html (Q1) and the decision log. If no-go → stop here; the task stays blocked/closed with the measurement recorded.
  2. T2 Config + deps + env var (REQ-020, NFR-002/004).
    GemsAcquisitionConfig (base URL, product, pinned ver, QA threshold, date range, batch window, AOI buffer, download budget, paths); document GEMS_API_KEY in .env.example; add data/raw/gems/ to .gitignore; add netCDF4 only if T1 showed the HDF layout needs it.
  3. T3 Open-API client: enumerate + download + auth (REQ-001, ERR-001/002).
    list_granules (getFileDateList/getFileList, pinned ver) + download_granule (getFileItem, key from env, retry/backoff, request-URL scrubbing); clear error referencing the env-var name on key failure; descriptive error on product/field-not-found.
  4. T4 Tests (mocked API + synthetic granule) for enumerate/download/clip/persist (REQ-001/005/006).
    Fake HTTP layer returns a listing + a small synthetic NetCDF granule covering the AOI with several scan slots; assert clip shape, fields retained, provenance written, full granule deleted after clip.
  5. T5 Client-side AOI clip + QA + per-scan write (REQ-001/004/005/006).
    Open granule lazily → bbox subset to AOI → QA mask → write per-scan NetCDF + provenance → delete the cached full granule (unless --keep-granules).
  6. T6 Diurnal preservation + coverage + version tracking (REQ-002/003/007).
    Every scan tagged with its observation timestamp (distinct slots/day); per-scan valid-coverage fraction over AOI; record processor version; flag version discontinuity across a range.
  7. T7 Resumable batching + scoped-window default + manifest + budget guard (REQ-008/009, ERR-003/004, NFR-005).
    Default scope = stable-weather autumn window Sep–Oct 2023 (config-overridable; optional Sep–Oct 2024 replication; full mission opt-in). Record per-scan calendar metadata (weekday + Chinese-holiday flag, esp. Golden Week 1–7 Oct). Manifest of fetched granules/scans; re-run skips done; batch by date window (SFTP bulk route; 20 GB/40-file cap is no-login-web only); rate-limit + backoff; stop + surface no-go if accumulated volume exceeds the budget.
  8. T8 Verification module (REQ-010/011/012).
    select_clear_high_scan; render_scan → NO₂-over-AOI PNG with facility overlay; render_diurnal_panel → AOI-mean NO₂ across the day's scans (the diurnal-resolution evidence).
  9. T9 Verification + error tests (AT5a, AT-ERR-1).

  10. T10 Wire CLI fetch-gems (+ --probe) + verify-gems (REQ-020) + smoke test.
    Config-driven subcommands; CLI smoke test (mocked).
  11. T11 Real acquisition slice + verification (AC-001/004). (developer-run on the server; needs the key)
