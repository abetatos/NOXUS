Deep-review record (high-risk block)

High-risk categories touched: large cross-cutting feature that revives a reverted module, statistical-claims surface (p-hacking risk across many tunable knobs), and a new external data dependency with an optional authenticated fallback (credential surface). Per the design risks section and reference/deep-review-policy.md.

Escalation rungs actually run

  • Rung 1 (always) — standard reviewer + High-risk checklist + security-focused pass: RUN. Manual statistical audit, credential-surface grep, git-tracking check, numeric reproduction of the core stats against scipy/hand-computation.
  • Rung 2 (recommended) — adversarial pass: RUN (single-reviewer, three lenses). Statistical-correctness lens (p-hacking, lag selection, null reachability), security lens (credentials, network in tests), correctness/edge lens (NaN/coverage, degenerate inputs). A separate independent reviewer subagent per lens was not spawned; this single deep pass exercised all three lenses adversarially and reproduced the headline claims numerically. Recorded as the accepted substitute.
  • Rung 3 (optional, paid) — cloud deep review / PR review service: NOT RUN. Requires explicit per-invocation developer approval and incurs cost; not requested. Its absence does not block per policy — rung 2 is the accepted substitute.

Statistical / p-hacking discipline

  • Null is genuinely reachable. test_planted_null_reports_null uses two independent random series (seeds 99) → |r|<0.3, p>0.05, sign indeterminate (never flipped, EDGE-007). The conclusion logic (_conclusion) returns "null" whenever the lag-0 correlation is not significant OR the peak |r| is below-bar — the null is not engineered away.
  • Planted signal recovers truth. seed-7 fixture, benchmark = 2·index(t−3) + noise → sign positive & significant, peak_lag == 3, r>0.5 p<0.05 on the lead-aligned series, CI reported. Reproduced independently.
  • Knobs are fixed in config and echoed. All researcher-degrees-of-freedom (deseason method, meteo form, lag window, cap, anchor, coverage floor, min_overlap, success bar) are SignalConfig/ValidationConfig defaults and are written into steel_validation_results.json config_echo and the plain-language summary. No silent defaults that could be tuned post-hoc without disclosure.
  • r/p/CI computed correctly. Verified numerically: correlate matches scipy.stats.pearsonr (modern API .statistic/.pvalue + confidence_interval) exactly; degenerate/zero-variance inputs return a safe null rather than crashing.

Blocking

Non-blocking (resolve or record explicit deferral before the whole task is done)

  • N1 — Curtailment provenance honesty (elevated: statistical-claims surface). _load_curtailment always returns None, so the curtailment exogenous control (REQ-022) is never applied in the end-to-end path, yet provenance records curtailment_source: "crea" and the summary echoes curtailment_control: crea. A reader would believe a curtailment control was applied when it was not. Reproduced: end-to-end build_activity_index path passes curtailment=None, so structural_terms contains only heating_season. Fix (small, low-risk): when no curtailment series is loaded, record curtailment_control: "absent (no calendar artifact; deferred to T15/Q6a)" (or surface structural_terms alongside the source) so the echo cannot overstate the control. Severity: medium (honesty/NFR-003). Does not block this slice; should be resolved (or explicitly deferred to T15 with the real curtailment calendar) before done.
  • N2 — Multiple-lag selection not disclosed (elevated: statistical-claims surface). lead_lag selects peak_r as the max |r| over 17 lags, and _conclusion classifies that peak against the success bar. The reported ccf_sig_bound (±1.96/√n) is an uncorrected per-lag white-noise band — selecting the maximum over many lags inflates the apparent peak significance. The headline significance gate correctly uses the lag-0 correlate p-value (not the peak), which mitigates the worst over-claim, but the inflation from peak selection is not surfaced. Theoretical (not a code bug); reproduced conceptually, not a failing test. Fix (doc-level): add a one-line note in render_summary / methods doc that the peak lag is selected over the lag window and the per-lag band is not multiplicity-corrected. Severity: low–medium. Resolve or defer with justification before done.
  • N3 — Summary n field provenance. render_summary reports n = r["n_overlap"] (= len(aligned)) rather than CorrResult.n (the jointly non-NaN count used by correlate). After align_series these are equal (it drops NaN), so the displayed n is correct today, but the CorrResult.n value is silently discarded. Fix (trivial, optional): carry corr.n into the results dict, or add an assertion that the two agree. Severity: low (cosmetic/robustness). Non-blocking.
