Summary

The exploration (docs/exploration-results.html) showed two things: (a) better seasonality removal helps — a single annual Fourier harmonic on the intensity residual lifts the best-regime coupling to r≈0.78 — and (b) the binding limit is the signal-to-noise: steel is only ~40% of the column mean, so a level estimator is capped near r≈0.63. The one lever that can raise steel's variance share without the (deferred) flux-divergence method is temporal source separation via the day-of-week cycle: steel runs 24/7 (baseload, flat across the week) while traffic/other sources carry a strong weekly cycle — so removing the weekly component isolates the industrial term. That cycle is invisible in the weekly composite and needs a daily series.

This feature (1) adds a daily NO₂ footprint series (aggregate the per-overpass data to a daily grid, coverage-screened, never interpolated) and (2) applies a Prophet decomposition — piecewise-linear trend (removes the secular emission- intensity decline) + yearly + weekly Fourier seasonality — to emit a deseasonalised activity residual and the day-of-week diagnostic, then compares it against the BF operating rate and re-runs the NOX-004 catalyst on the cleaner residual.

Feasibility (verified 2026-06-13): 2,617 overpasses span 2019-01-01→present; the daily footprint has 1,984 valid days (73%) after cloud screening, and valid days are balanced across weekdays (Mon 290 … Sun 270) — so the weekly cycle is observable, not cloud-aliased. prophet 1.3.0 is installed (developer-approved).

Honesty (carried over): the SNR ceiling is real — the 40% mean share bounds a level estimator but steel's variance share rises to ~60% when curtailments are the swing factor, so the catalyst/event framing remains the right consumer. Strict no look-ahead for any predictive/market claim (Prophet refit causally), and an honest null is a valid outcome (Morris & Zhang 2019).

Functional requirements

EARS format. IDs are stable references for tasks and tests.

Daily NO₂ aggregation

REQ-001When the daily path runs, the system shall composite the per-overpass NO₂ store (NOX-002a, data/raw/tropomi/) to a daily cube using the NOX-002b gridding at freq="D", applying the inherited coverage screening so cloudy/empty days are masked (NaN), never interpolated (NOX-002b cloud decision).
REQ-002When sampling, the system shall produce a daily footprint NO₂ series (NOX-003 footprint sampling + background correction at daily resolution) with per-day valid_coverage, and shall record the valid-day fraction and the per-weekday valid-count so weekday balance is auditable (EDGE-002).

Prophet decomposition

REQ-010When deseasonalising, the system shall fit a Prophet model on the daily footprint NO₂ with a trend (piecewise-linear, changepoints) + yearly + weekly Fourier seasonality, fitting only on observed (non-NaN) days (Prophet handles the irregular/gappy index natively), and shall emit the trend, yearly, weekly components and the deseasonalised residual (activity proxy) on the observed days only.
REQ-011When fitting, the system shall be deterministic (MAP estimation; uncertainty_samples=0 or a fixed seed) so a given input + config reproduces the same decomposition (NFR-005), and shall record the Prophet config (growth, changepoint prior, yearly/weekly Fourier orders) in the run provenance.
REQ-012When emitting the residual, the system shall return it only on observed days — Prophet's fitted value on a cloud-gap day is a model interpolation and shall not be presented as an observation (no fabricated activity, EDGE-001).

Weekday source-separation diagnostic

REQ-020When the weekly component is available, the system shall emit a day-of-week source-separation diagnostic: the weekly seasonal shape and its amplitude, interpreted as — a flat weekly cycle ⇒ steel-dominated (baseload); a strong weekday/weekend cycle ⇒ a non-steel (traffic/other) component now removed from the activity residual. The diagnostic shall quantify the variance removed by the weekly term.
REQ-021When reporting, the system shall test the hypothesis that removing the weekly cycle raises steel's effective variance share — i.e. whether the Prophet residual couples to the BF rate better than the weekly intensity/harmonic residual (NOX-003.1) — and shall report the result honestly, including a null.

Comparison & integration

REQ-030When comparing, the system shall correlate the daily-Prophet residual (and its weekly/monthly aggregates) against the CREA BF operating rate with the established metrics — full-sample r, rolling-regime peak, and lead-lag — alongside the existing yoy / STL / intensity / harmonic residuals, so the marginal value of the daily-Prophet path is explicit.
REQ-031When integrated, the system shall expose the Prophet decomposition as a selectable deseason_method="prophet" (additive; existing methods unchanged) and shall allow the NOX-004 catalyst to re-run on the daily-Prophet residual, reporting whether event precision/recall/lead improve.
REQ-032When any lead or market claim is made from the Prophet residual, the system shall use a causal decomposition (Prophet refit on data up to each decision time, or an equivalent rolling refit) — the in-sample full-series fit is for diagnosis only, never for a tradeable lead (no look-ahead, carrying NOX-004 REQ-041).

CLI / orchestration

REQ-040When the user runs the daily path via the noxus CLI, the system shall expose it through documented config/subcommands (e.g. a daily grid/attribute mode + a deseason_method="prophet"), reproducible from a recorded config; existing weekly subcommands and defaults are unchanged.

Non-functional requirements

NFR-001Reproducibility. Daily cube + Prophet residual reproduce from the committed/derived inputs + recorded config; no live fetch at analysis time.
NFR-002Public/free data only. Same TROPOMI store, re-aggregated; prophet is a free OSS dependency (developer-approved).
NFR-003No look-ahead for predictive use. In-sample Prophet is diagnostic; a causal/rolling refit is required before any lead/market claim.
NFR-004Honest reporting. Report the marginal value (or null) of the daily-Prophet path with the SNR-ceiling caveat; no cherry-picked window.
NFR-005Determinism. Prophet in MAP mode (uncertainty_samples=0); fixed seeds; identical inputs → identical outputs.
NFR-006No-publish. Daily NO₂-derived series (cube, footprint, residual) are gitignored; small stats summaries + figures may be committed.
NFR-007Stack conformance. Python ≥3.12; passes uv run pytest + uv run ruff check .; prophet lazy-imported so the rest of the package loads without it and tests mock/skip it.

Edge cases

EDGE-001Long cloud gap → Prophet still fits trend/seasonality across it, but the residual is emitted only on observed days; the interpolated fit is never presented as data.
EDGE-002Weekday-biased cloud cover would alias the weekly term → the valid-per-weekday counts are reported; if badly imbalanced, the weekly component is flagged unreliable (verified balanced 2026-06-13).
EDGE-003Two overpasses on one day (orbit overlap at the AOI edge) → daily compositing means over valid cells, de-duplicated, not double-counted.
EDGE-004Prophet changepoint over-fitting (trend chases noise) → constrain the changepoint prior and report sensitivity, mirroring the NOX-003.1 smoothness discipline.
EDGE-005Partial cube (fetch incomplete) → the daily path still runs and labels the result partial; the full-fetch re-run remains the arbiter (NOX-003.1 T10).

Error states

ERR-001If the per-overpass store / manifest is missing, the daily compositing fails with an actionable message naming noxus fetch.
ERR-002If prophet is not installed, the Prophet path fails with an actionable install message (and the harmonic method remains available as the dependency-free fallback).
ERR-003If fewer than a configured minimum of valid days exist, the system refuses the Prophet fit and reports insufficient data rather than over-fitting.

Acceptance criteria

AC-001The daily compositing produces a daily footprint series with coverage screening; cloud-gap days are NaN (not interpolated); valid-day fraction + per-weekday counts are emitted (REQ-001/002).
AC-002On a synthetic daily series (planted trend + yearly + weekly + activity + noise) the Prophet decomposition recovers the trend and the weekly/yearly components within tolerance, and the residual recovers the planted activity; the residual is emitted only on observed days (REQ-010/012).
AC-003Determinism: refitting on identical input/config yields an identical residual (NFR-005).
AC-004The weekday diagnostic recovers a planted weekly cycle's amplitude and reports the variance it removes; a flat-weekly synthetic yields ~zero weekly amplitude (REQ-020).
AC-005The comparison table reports the daily-Prophet residual vs BF (full + rolling-regime + lead-lag) alongside yoy/STL/intensity/harmonic; the catalyst re-runs on the Prophet residual (REQ-030/031).
AC-006A causal/rolling-refit path exists and a guard test shows the residual at day t used for any lead claim does not depend on future days (REQ-032, NFR-003).
AC-007prophet is lazy-imported (package imports without it); ERR-002 fires with an actionable message when absent; harmonic fallback works (REQ ERR-002).
AC-008uv run pytest and uv run ruff check . pass.
